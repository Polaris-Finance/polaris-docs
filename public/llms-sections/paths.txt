# Develop on Polaris

**Who this is for:** engineers building an app, indexer, risk system, vault, liquidation monitor, analytics surface, or external market around Polaris.

**Prerequisites:** familiarity with onchain reads/writes (viem, ethers, or equivalent) and the Polaris product model. New to these concepts? Read [Core Concepts](https://tokenbrice.github.io/polaris-docs/core-concepts) first.

> **Launch status:** Polaris is in public Public Testnet 1 on Sepolia, with production mainnet forthcoming. Contract addresses, ABIs, verified source, canonical events, SDKs, subgraphs, and API endpoints for production are not yet published. Build against the documented model, but do not hardcode pre-launch addresses, method names, or event signatures. Track [Polaris Launch Status](https://tokenbrice.github.io/polaris-docs/launch-status) before binding to artifacts.

## Reading sequence

Read these in order — model the system before you bind to any artifact.

1. [Core Concepts](https://tokenbrice.github.io/polaris-docs/core-concepts) — the system model and its component boundaries.
2. [Contracts and Addresses](https://tokenbrice.github.io/polaris-docs/resources/contracts) — the deployment shape (a shared core plus one instance per pAsset), the contract set, and the address, source, and ABI format to verify against.
3. [Audits and Security](https://tokenbrice.github.io/polaris-docs/resources/audits-security) — the security model and audit status; read before treating a deployment as production-grade.
4. [Polaris Launch Status](https://tokenbrice.github.io/polaris-docs/launch-status) and [Safety and Verification](https://tokenbrice.github.io/polaris-docs/paths/safety-verification) — the pre-interaction checklist for network, chain ID, addresses, and verified source.

## Build now, bind later

Until deployment artifacts ship, the safest useful work is interface planning:

- Model the product concepts: pETH, pAssets, troves, Stability Pools, Fee Router, POLAR, and vePOLAR.
- Design data schemas around stable conceptual entities, not unpublished function names.
- Build verification gates so your app refuses unverified networks or addresses.
- Keep chain-specific config external and replaceable.
- Track the official docs for the deployment manifest, ABIs, and event catalog.

## Gotchas

- **Testnet addresses are not production addresses.** Treat Sepolia manifests as replaceable fixtures. Never use them as defaults for mainnet.
- **One pAsset means one instance.** A pUSD trove and a pGOLD trove are unrelated positions with different managers, pools, parameters, and risk states.
- **pETH is collateral, not pAsset debt.** Do not put pETH in the same accounting bucket as CDP-minted pAssets.
- **Redemptions are global to an instance.** There is no sorted lowest-ICR queue. If you model borrower exposure, model pro-rata reduction across all open troves for that pAsset.
- **Fee Router pETH rewards and primary-interest splits are separate streams.** Do not assume every yield destination is paid by the same contract or denomination.
- **Final event names are pending.** Build your indexer so event bindings are generated from the published ABI, not copied from draft examples.

## Failure modes to handle

Failure mode | Integration response

Wrong chain ID | Refuse reads/writes and show the expected chain from [Launch Status](https://tokenbrice.github.io/polaris-docs/launch-status).
Address missing from the manifest | Disable the feature rather than falling back to a search result or environment default.
Explorer source or ABI mismatch | Treat the deployment as unverified and fail closed.
Stale oracle or unavailable price path | Hide leverage/borrowing actions and show risk state as unknown.
Recovery Mode active | Block or warn on collateral-reducing and debt-increasing flows according to live contract state.
Redemption, liquidation, or conversion quote changes before inclusion | Requote before signing and use explicit user slippage bounds.
Unknown pAsset instance | Do not share pUSD assumptions with another pAsset; load per-instance parameters.

## Worked integration sketch

The exact package, ABI, and method names are pending. The important pattern is a fail-closed deployment gate before any read or write.

This is integration pseudo-code, not a published SDK. The shape is copy-safe TypeScript, but the explorer-verification implementation must come from your own stack or a future Polaris SDK after canonical ABIs publish.

```ts
type Address = `0x${string}`

type PolarisDeployment = {
  chainId: number
  contracts: Record<string, Address>
  abiHashes: Record<string, string>
}

type ReadClient = {
  getBytecode(args: { address: Address }): Promise<`0x${string}` | null | undefined>
}

type ExplorerVerifier = (args: {
  address: Address
  expectedAbiHash: string
}) => Promise<boolean>

const EXPECTED_CHAIN_ID = 11155111 // Public Testnet 1 today; production mainnet is 1 after launch.
```

```ts

  client: ReadClient,
  deployment: PolarisDeployment,
  verifyExplorerSourceAndAbiHash: ExplorerVerifier
) {
  if (deployment.chainId !== EXPECTED_CHAIN_ID) {
    throw new Error(`Unexpected Polaris network: ${deployment.chainId}`)
  }

  for (const [name, address] of Object.entries(deployment.contracts)) {
    const code = await client.getBytecode({ address })
    if (!code) throw new Error(`Missing bytecode for ${name}`)

    const verified = await verifyExplorerSourceAndAbiHash({
      address,
      expectedAbiHash: deployment.abiHashes[name]
    })
    if (!verified) throw new Error(`Unverified Polaris contract: ${name}`)
  }

  return {
    peth: deployment.contracts.peth,
    pUSDManager: deployment.contracts.pUSDManager,
    pUSDStabilityPool: deployment.contracts.pUSDStabilityPool
  }
}
```

Use the same shape for production later, but with Ethereum mainnet chain ID `1` and only after [Contracts & Addresses](https://tokenbrice.github.io/polaris-docs/resources/contracts) publishes canonical production artifacts.

# Polaris Safety and Verification

**Who this is for:** every Polaris user and integrator; run this path before connecting a wallet, sending ETH, minting a pAsset, depositing into a pool, locking POLAR, or integrating contracts.

**Prerequisites:** none. Complete this path *first*, before any other journey, because impersonation and phishing are most likely around launch and testnet phases.

> **Launch status:** [Polaris Launch Status](https://tokenbrice.github.io/polaris-docs/launch-status) is the source of truth. As of June 8, 2026, public Public Testnet 1 is on Sepolia (`11155111`) and the official testnet app is [app.testnet.polarisfinance.io](https://app.testnet.polarisfinance.io/). Production Ethereum mainnet (`1`) is forthcoming, and no production app, production addresses, ABIs, verified source, final parameters, or audit reports are published yet.

> **Open Testnet App:** Use [https://app.testnet.polarisfinance.io/](https://app.testnet.polarisfinance.io/) only on Sepolia (`11155111`) with no-value test assets. Treat any production frontend, token, contract, airdrop, or support message as unverified until it is linked from [Launch Status](https://tokenbrice.github.io/polaris-docs/launch-status).

App anchors: **Dashboard** first for network and system state, then any action tab's pre-confirmation screen for contract, token, amount, fee, approval, and simulation checks.

## Verification checklist

Work top to bottom; each row names a check and where it will be confirmed at launch.

Check | Where it will be confirmed

Current launch phase and network | [Polaris Launch Status](https://tokenbrice.github.io/polaris-docs/launch-status)
Testnet app URL and production app status | [Polaris Launch Status](https://tokenbrice.github.io/polaris-docs/launch-status) and [Getting Started](https://tokenbrice.github.io/polaris-docs/getting-started)
Deployed addresses | [Launch Status](https://tokenbrice.github.io/polaris-docs/launch-status), official Sepolia manifests, and [Contracts and Addresses](https://tokenbrice.github.io/polaris-docs/resources/contracts) once production is live
Verified source and ABIs | [Contracts and Addresses](https://tokenbrice.github.io/polaris-docs/resources/contracts), official manifests, and block explorer links
Audit reports | [Audits and Security](https://tokenbrice.github.io/polaris-docs/resources/audits-security)
Final user parameters | The relevant product page and contract reference
Stewardship bounds | [Stewardship, not Governance](https://tokenbrice.github.io/polaris-docs/stewardship/stewardship)

## Things Polaris will not ask for

- A seed phrase or private key.
- KYC for protocol use.
- A direct transfer to a support wallet.
- Interaction with an unpublished address.
- Trust in a frontend that is not linked from official Polaris channels.

## Developer verification

For apps and indexers, add explicit allowlists for network IDs and contract addresses. Testnet allowlists should point to Sepolia (`11155111`) artifacts from [Launch Status](https://tokenbrice.github.io/polaris-docs/launch-status) and the linked manifests. Production allowlists should stay empty until production addresses are published. Your integration should fail closed if the deployment manifest, ABI, or verified source does not match the official docs.
